Figma 治理规约

从"自由画布"到"工程声明文件"

🧠 核心共识

"AI 不看像素，看的是图层树"

设计稿的深度决定了代码的高度。如果设计师画出来的只是"一张图"，那 AI 给你的是"一段代码"；
如果是一套"结构"，那 AI 给你的是"一个可以直接上线的工程"。

1

结构卫生

图层树即 DOM 树，混乱的图层 = 混乱的代码

🚫 禁止

语义化命名

❌ 错误示例

Frame 1024, Group 47, Rectangle 89

✅ 正确示例

Header, NavBar, UserAvatar, LoginForm

必须使用语义化英文命名，让 AI 能理解每个图层的业务含义。

⚠️ 应当

消除冗余层级

尽量减少不必要的嵌套，能用一层 Auto Layout 解决的绝不包两层 Frame。
过深的嵌套会导致 AI 生成冗余的 <div> 包裹。

💡 建议

Slot 占位逻辑

对内容可变区域使用"占位组件"，并在命名中标记（如 Slot_ContentArea），方便 AI 生成可替换的代码逻辑。

2

物理约束：强制 Auto Layout

Figma Auto Layout = CSS Flexbox，没有 Auto Layout 只能用 absolute 定位

若不设 Auto Layout，AI 只能用 position: absolute 定位，导致代码难以维护、无法响应式。

Figma

Fill Container

→

CSS

flex: 1

Figma

Hug Contents

→

CSS

width: fit-content

⚠️ 警告：没有 Auto Layout 的 Frame 将被 AI 识别为"自由定位容器"，生成大量 top: 47px; left: 128px; 这样的硬编码样式。

3

组件主权

白名单机制 + 禁止 Detach，确保组件来源可控

🔒 强制

白名单机制

AI 仅识别来自指定组件库（如 TAI_UI_Library）的组件。
非白名单组件将被标记为"未知元素"，需人工确认。

🚫 禁止

禁止 Detach（打散）

❌ 绝对禁止私自打散组件后手动修改，否则 AI 会将其视为"脏数据"，丢失组件语义信息。

💡 建议

参数映射 (Properties → Props)

利用 Figma 的 Properties 功能定义组件变体，让 AI 能直接生成对应的组件 Props。

Figma Property

variant="primary"
→

React Props

<Button variant="primary">
4

变量同步

Figma Variables = tokens.json，设计与代码数据源对齐

🔒 强制

全量引用变量

颜色、圆角、间距等必须使用 Figma Variables，实现与代码侧（tokens.json）的数据源对齐。

❌ 硬编码

#3B82F6

✅ 引用变量

$color-primary-500

MCP 互传协议

确保设计稿与代码库之间的变量修改能双向同步：
