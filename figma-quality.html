Figma 治理规约

从"自由画布"到"工程声明文件" — 让设计稿成为 AI 的可执行蓝图

核心共识
🎨 AI 不看像素，看的是图层树

设计稿的深度决定了代码的高度。如果设计师画出来的只是"一张图"，那 AI 给你的是"一段代码"； 如果是一套"结构"，那 AI 给你的是"一个可以直接上线的工程"。

1

结构卫生

图层树即 DOM 树 — 混乱的图层 = 混乱的代码

🚫 语义化命名（强制）

禁止无意义命名，必须使用语义化英文命名，让 AI 能理解每个图层的业务含义。

❌ 错误
Frame 1024
Group 47
Rectangle 89
✅ 正确
Header
NavBar
UserAvatar

⚠️ 消除冗余层级

尽量减少不必要的嵌套，能用一层 Auto Layout 解决的绝不包两层 Frame。 过深的嵌套会导致 AI 生成冗余的 <div> 包裹。

💡 Slot 占位逻辑

对内容可变区域使用"占位组件"，并在命名中标记（如 Slot_ContentArea），方便 AI 生成可替换的代码逻辑。

📐 图层树 ≈ DOM 树

保持视觉层级与代码结构一致。图层树的父子关系应直接映射为 HTML 的嵌套关系。

2

物理约束：强制 Auto Layout

Figma Auto Layout = CSS Flexbox — 没 Auto Layout 就等着 absolute 定位吧

🔗 映射关系

明确 Figma 布局属性到 CSS 的对应关系：

Figma 属性 CSS 等价
Auto Layout → display: flex
Fill Container → flex: 1
Hug Contents → width: fit-content
Fixed → width: 320px（硬编码）

⚠️ 警告：没有 Auto Layout 的 Frame 将被 AI 识别为"自由定位容器"，生成大量 top: 47px; left: 128px; 这样的硬编码样式。

3

组件主权

白名单机制 + 禁止 Detach — 确保组件来源可控

🔒 白名单机制（强制）

AI 仅识别来自指定组件库（如 TAI_UI_Library）的组件。非白名单组件将被标记为"未知元素"，需人工确认。

🚫 禁止 Detach（打散）

❌ 绝对禁止私自打散组件后手动修改。AI 会将打散后的图层视为"脏数据"，丢失组件语义信息，无法映射到代码组件库。

💡 参数映射 (Properties → Props)

利用 Figma 的 Properties 功能定义组件变体，让 AI 能直接生成对应的组件 Props。

Figma Property React Props
variant = "primary" → <Button variant="primary">
size = "large" → <Button size="lg">
disabled = true → <Button disabled>
4

变量同步

Figma Variables = tokens.json — 设计与代码数据源对齐

🔒 全量引用变量（强制）

颜色、圆角、间距等必须使用 Figma Variables，实现与代码侧（tokens.json）的数据源对齐。

❌ 硬编码
#3B82F6
border-radius: 8px
padding: 16px
✅ 引用变量
$color-primary-500
$radius-md
$space-4

🔄 MCP 互传协议

确保设计稿与代码库之间的变量修改能双向同步：

🎨 Figma 修改变量
→
🔄 MCP 同步
→
💻 tokens.json 更新
→
🏗️ 代码变量自动更新

🛠️ 团队治理方案

在 Figma 转代码之前，通过自动化审计拦截不合规的设计稿，建立设计 → 代码的闭环工作流。

🔍 自动化 Lint 审计

在转代码前自动检查以下项目，不合格的设计稿直接拦截：

  • 图层命名规范 — 是否使用语义化英文命名
  • 间距使用变量 — 是否引用 Figma Variables 而非硬编码
  • 颜色引用检查 — 是否有未绑定变量的硬编码色值
  • 组件来源白名单 — 是否全部来自指定组件库
  • 图层树深度 — 嵌套是否超过 5 层

🔄 Figma ↔ MCP 协作循环

建立"设计输出 → AI 生成 → 代码精修 → 反向回写"的闭环工作流：

🎨 设计输出
→
🤖 AI 生成代码
→
👨‍💻 代码精修
→
↩️ 反向回写设计稿

✅ Figma 交付前检查清单
