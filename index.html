1. 现象分析：理想与现实的鸿沟

💭 理想主义 (Hype)

  • ❌ "一句话就能生成完整的商业应用。"
  • ❌ "AI能和设计师心灵相通，理解模糊的需求。"
  • ❌ "以后不需要前端开发和交互设计师了。"
  • ❌ 对着界面说"给我加个AI入口"，它就能完美契合业务流。

🧱 工程现实 (Reality)

  • ✅ AI本质是概率预测引擎，缺乏业务上下文，需要显式喂养。
  • ✅ 模糊的提示词必定产生随机的结果（Hallucination）。
  • ✅ 它的定位是"代码/组件装配工"，而非产品架构师。
  • ✅ 必须将感性设计转化为理性协议，AI 才能真正落地。

2. 底层逻辑：AI 是如何运作的

📐 概率预测与自回归机制的本质

大型语言模型（LLM）在生成代码时，其核心是一个基于概率的序列预测器。对于任何给定的输入序列，模型实际上是在计算下一个标记（Token）的条件概率分布：

P(xt+1 | x1, x2, ..., xt; θ)

其中 θ 代表模型的参数，x 代表输入的 Token 序列

⚠️ 关键认知：这意味着 AI 并不理解什么是"美学"或"品牌一致性"，它只是根据训练数据中代码与描述的统计关联，预测出最可能出现的代码片段。当用户提出"增加一个AI入口"时，模型可能会在导航栏添加图标，但它无法理解该入口背后的交互逻辑、权限控制或数据流向。

从 LLM 到 MLLM —— 多了一双"眼睛"

以上是所有大语言模型（LLM）的共同底座：基于文本 Token 的自回归概率预测。当模型进化为多模态版本（MLLM）后，它在文本预测能力之上增加了一个视觉编码器（通常是 ViT）——能"看"图了，但看的方式和人完全不同：它把图片切成若干小补丁（Patch），编码为 Token，再和文字 Token 一起送进 Transformer 做概率预测。这意味着它有视觉输入能力，但没有空间坐标系，间距、对齐、层级关系仍然靠"猜"。

MLLM (多模态大语言模型)

本质：基于上下文的 Token 概率预测

👁️

它如何"看"UI？

AI 并不具有人类的空间视觉。多模态大模型会将图像切割、编码转化为文本语义。

边界：对像素级对齐、微小间距、复杂的绝对定位理解偏差

🧠

它如何"思考"设计？

它通过学习 GitHub 上大量的 HTML/CSS/React 代码模式来"拼凑"UI。它不懂业务，只懂"这种代码模式通常连着那种模式"。

边界：缺乏真正的用户同理心和复杂的交互逻辑推导。

🗣️

它如何"听"需求？

强依赖于 Context Window（上下文窗口）。"加个AI入口"，如果没给它当前的 DOM 系统规范，它只能靠猜。

边界：需要极其精确的 Prompt（System & 约束）。

为什么人类是"空间天才"，而 AI 是"空间路痴"？

一句话总结：人类脑子里自带坐标纸，AI 没有。所以 AI 嘴里说着"居中对齐"，写出来的代码里按钮可能偏了 3px 或者盖住了文字。这就是所谓的"接地（Grounding）局限"。

🧠 人类大脑：自带 GPS

格点细胞 — 你脑子里的"经纬网"

闭上眼在房间里走五步再转身，你依然能指出门口在哪。因为大脑的内嗅皮层有格点细胞，像蜂窝一样在脑海里铺了一张六边形虚拟网格，你每移动一步，它就像 GPS 信号一样告诉你当前坐标。

海马体 — 你脑子里的"活地图"

海马体把格点细胞的坐标存成记忆。看一眼 Figma 稿，你自动记住"左上角有 Logo，右边有导航，中间隔了一段距离"——不仅记住了有什么，还记住了它们的物理方位。

✅ 结果：人类看 UI 时，脑子里自动给界面打上了 1280×800 的底稿，对"左边一点""对齐""居中"有天然的物理直觉。

🤖 AI 大模型：只有记忆，没有坐标纸

它怎么"看"图？— 切碎拼猜

MLLM 看图不是用"眼睛"扫描，而是把图片切成一堆碎块（Patch）。它能认出这块碎片里有"按钮"，那块碎片里有"文本"，但只能通过统计概率猜出它们通常在一起。

致命伤 — 脑子里没有坐标纸

当它把碎块拼成代码时，它不知道坐标 (10, 20) 和 (10, 25) 之间真实的视觉差距有多大。它手里的"笔"是不受"经纬网"控制的。

❌ 结果：生成的界面经常出现元素重叠、间距不均、按钮偏移、比例失调等问题。

💡 实战启示：既然 AI 脑子里没坐标，我们就给它"外挂一套经纬网"

AI 的缺陷（无格点细胞） 表现 我们的工程补救
空间感模糊 分不清 8px 和 16px 的区别 注入 tokens.json → 给它"固定刻度"
逻辑断层 知道要加按钮，不知道加在哪最顺手 注入 spec.md → 用文字强制定义方位
比例失调 生出来的 Demo 忽大忽小 注入封装代码块 → 给它零件去"摆积木"

🎯 核心结论：我们不能让 AI 去"画"像素，我们要让 AI 按照我们给它的坐标系去"摆"积木。

认知维度对比：人类设计师 vs 生成式 AI

认知维度 人类设计师 生成式 AI (MLLM)

3. 能力边界分析：人机协同的界限

🤖 AI 核心优势：

极速原型生成：秒级产出讨论物料；标准组件装配：完美遵循组件库规范；多版本样式探索：基于风格迁移的海量方案出图。

👤 人类核心职责：

业务意图抽象：定义"为什么"；设计协议制定：输出 JSON/MD 物理规则；复杂逻辑闭环：API 绑定与状态管理；最终品控：审美最终决策。

4. 规格驱动开发 (SDD)：让人话也能产出工程级代码

范式转移：不是改变用户说话的方式，而是升级 AI 的"耳朵"

一个常见的误解是：要用好 AI，就必须学会说"机器话"。实际上，用户该怎么说话还怎么说话，"帮我做一个登录页面"完全没问题。真正要改变的不是用户的嘴，而是AI 接收信息的基础设施——通过预配置好的工程底座（System Prompt、规格模板、令牌体系），让系统在背后自动把人话翻译成机器可读的契约。

这就是"规格驱动开发（Spec-Driven Development, SDD）"的本质：规格注入应该是透明的、自动化的、预配置的，而不是要求每个人都变成提示词工程师。

Level 1

裸聊模式

用户说的

"帮我做个登录页面"

背后发生的

什么都没注入，AI 纯靠猜

产出质量

❌ 随机、不可控、每次结果都不同

Level 2

半武装模式

用户说的

"帮我做个登录页面"

背后发生的

手动贴了 tokens + 一些约束，但每次都要重复操作

产出质量

⚠️ 有改善但费力，依赖个人经验

Level 3

全武装模式 (SDD)

用户说的

"帮我做个登录页面" — 同样的话

背后发生的

System Prompt 自动注入 spec 模板 + tokens.json + 组件库索引

产出质量

✅ 可控、可审计、可复现

💡 三个 Level 里用户说的话完全一样。差别不在于"嘴"，而在于 AI 的"耳朵"——背后预配置了多少工程底座。

🔌 Context Injection Skill

可封装为 Skill — 用户还是说人话，Skill 在背后帮他穿上盔甲：自动注入 spec 模板 + tokens.json + 组件库索引。

🏗️ 设计系统作为"地面真实"：三层令牌体系

AI不应被允许"发明"新组件，而应被约束在现有的设计系统框架内运作。通过建立三层令牌体系，AI的职能从"创意绘图"转变为"组件组装"。

💡 注意：以下三层令牌体系就是上面 Level 3「全武装模式」中，系统在背后自动注入给 AI 的核心基础设施之一。用户不需要记住这些，也不需要在对话中提及它们——它们由团队的工程底座预先配置好，AI 会话启动时自动加载。你的同事只需说"帮我做个登录页"，这套令牌体系就已经在背后默默约束着 AI 的每一行输出。

原始令牌 (Primitive Tokens)

Layer 1
描述：基础数值定义
示例：#0052CC, 8px
意义：防止 AI 随意使用十六进制色值

语义令牌 (Semantic Tokens)

Layer 2
描述：描述用途的名称
示例：--color-button-primary
意义：传达"为什么"使用该颜色，而非仅"是什么"

组件令牌 (Component Tokens)

Layer 3
描述：绑定到特定 UI 元素
示例：--button-large-padding
意义：提供精确的样式约束，杜绝随机输出
🔌 Design Token Skill

可封装为 Skill — AI 写代码时自动带着调色盘和尺子：三层令牌体系作为 Skill 知识库，实时校验颜色/间距/圆角是否合规。

5. 工业级标准化生产规程

将模糊对话转化为**机器可执行的协议**。这是确保 AI 产出"工程级代码"的唯一路径。

1

样式协议化 (Skeleton)

tokens.json

定义物理约束。建立三层令牌体系（原始值 → 语义化 → 组件级），消除 AI 的视觉盲猜，让颜色、间距、字号全部有据可依。

📑 深度示例：语义化 Token 体系
{ "project": "TAI-6.0", "primitive": { "teal-500": "#0D9488", "teal-700": "#0F766E", "stone-900": "#1C1917" }, "semantic": { "background": { "main": "{primitive.stone-900}", "action": "{primitive.teal-500}", "action-hover": "{primitive.teal-700}" }, "text": { "on-action": "#FFFFFF", "muted": "#A8A29E" } }, "layout": { "radius": { "card": "24px", "btn": "12px" }, "spacing": { "md": "16px", "lg": "32px" } }, "component-constraints": { "primary-button": { "min-height": "64px", "font-weight": "700" } } }
🔌 Design Token Skill

此处即模块 4 中 Design Token Skill 的工程落地 — 这份 tokens.json 就是 Skill 知识库的核心载体。

2

意图规格书 (Soul)

spec.md

注入设计的灵魂。用结构化文档解释业务场景逻辑与"不可逾越的红线"。采用会话拆分策略：基础加载 → 分步生成 → 上下文回填。

📝 深度示例：包含状态机逻辑的交互规格书
# 场景：驾驶模式下的智能助手交互 ## 1. 核心触发逻辑 - **初始化**：组件以 `semantic.background.main` 色彩常驻于屏幕底栏 20% 处。 - **驾驶感应**：若检测到 `car_speed > 0`，强制进入"安全性门禁模式"： - 自动将按钮点击热区扩大至全宽。 - 禁止展示任何细颗粒度的辅助文本，仅保留核心 Action。 ## 2. 反馈闭环规范 - **监听中 (Listening)**：背景需执行 2.0s 周期的渐变呼吸效果。 - **思考中 (Thinking)**：展示 `tokens.json` 中定义的 `action` 颜色的 loading 状态。 - **错误边界**：若 AI 响应超时，展示警告图标并回退至上一级导航，不得在当前页死等。
🔌 Spec Template Skill

可封装为 Skill — 告诉 AI 你是造车的还是做后台的，它自动切换脑子：根据项目类型匹配对应的 spec 模板，注入交互状态机和红线约束。

3

封装参考库 (Reference)

.tsx / .jsx
核心资产库

这是存放你封装代码块的地方！ 喂入私有源码（Few-shot），AI 才能生成真正工程化的调用。

💻 深度示例：企业级封装组件源码 (带业务 Hook)
// 你已经写好的私有资产：TAI_BaseCard.tsx import { useDriverState } from '@/hooks/useDriverState'; import { Tokens } from '@/config/theme'; export const TAI_Container = ({ title, children, status = 'idle' }) => { const { isDrivingMode } = useDriverState(); return ( <div className={`p-6 flex flex-col gap-4 rounded-[--radius-card] shadow-xl ${isDrivingMode ? 'bg-secondary' : 'bg-primary'} ${status === 'error' ? 'border-red-500' : 'border-stone-800'}`}> <div className="flex justify-between items-center"> <h3 className="text-xl font-extrabold text-white">{title}</h3> <div className="status-indicator" /> </div> <div className="w-full h-px bg-stone-700 opacity-30" /> <div className="card-body-content"> {children} </div> </div> ); };
🔌 Component Library Skill

可封装为 Skill — AI 从你的零件箱里拿积木，不是自己捏泥巴：将私有组件库索引为 Skill 知识库，AI 生成代码时检索已有封装而非"发明新的"。

3+

状态管理与 API 绑定 (Logic Bridge)

进阶

AI 在生成 React Hooks 或复杂状态机时经常产生幻觉，导致竞态条件。生产规程应强制要求使用确定性的状态模型。

维度 AI 生成的原始逻辑 工业级规程要求
状态表示 散乱的 useState (isLoading, isError) 集中式的 XState 状态机模型
状态转移 隐式的副作用调用 显式的事件驱动转移 on SUCCESS -> 'loggedIn'
API 交互 伪造的 Mock 数据或路径 基于 OpenAPI/Swagger 规格的类型安全调用
🔌 State Machine Skill

可封装为 Skill — 不让 AI 乱接线，给它一张标准电路图：封装 XState 状态机模板 + OpenAPI 规格解析能力，强制 AI 走确定性状态模型。

4

人机交接与交付 (QA Gate)

Delivery

明确工程边界：AI 产出"皮囊"，人类负责注入"数据活水"。

🤖 AI 产出的 80%

  • ✔ 复用 Step 3 里的私有封装组件进行装配
  • ✔ 自动填入 Step 1 里的语义化变量名
  • ✔ 生成响应式布局代码及 UI 状态展示逻辑
  • ✔ 遵循 Step 2 的交互场景物理约束

👤 人类介入的 20%

  • ✔ 后端 API 数据真实接口联调与错误分支捕获
  • ✔ 全局 Context/Redux 状态树同步与权限分发
  • ✔ 针对极细微交互细腻度的最后像素级微调
  • ✔ 性能优化及正式部署前的工程安全性审计

6. 人机协同：六个强制性检查点 (HITL)

通过构建"人机回环（Human-in-the-Loop）"系统，在 AI 的自主性中嵌入人工审慎，将风险降至最低。

1

🚦 需求解析门禁 (Requirement Gate)

在 AI 开始写代码之前，人类必须通过"提示词规格书"进行签名确认。

触发条件：AI 初步语义分析置信度 < 阈值 执行：自动挂起，请求人类提供更多上下文
2

🔍 组件映射审计 (Component Audit)

利用自动化脚本（如 token-audit.js）扫描 AI 计划使用的组件和令牌。

拦截：硬编码十六进制颜色 拦截：未定义的 CSS 类 引导：重新参考设计系统 MD 文件
🔌 Token Audit Skill

可封装为 Skill — AI 写完代码自动跑一遍质检：将 token-audit.js 封装为 Skill，每次生成后自动扫描硬编码颜色、未定义 CSS 类和不合规调用。

3

⏸️ 实时交互中断 (Signal Interrupts)

利用 Temporal 等工作流引擎，将 AI 生成过程设计为可中断的任务节点。

信号：UserDecisionSignal (保留/修改/等待) 防止：在错误路径上生成过长无效代码
4

🧪 四层逻辑一致性校验

AI 生成的代码必须通过四层验证策略：

L1: XML/HTML 语法完整性校验
L2: XSD / TypeScript 类型架构验证
L3: 属性放置检查（按钮属性误植文本框）
L4: 业务版本兼容性（React 18 并发渲染）
5

👁️ 视觉回归测试 (AVRT)

生成的界面必须在多个模拟视口中进行自动化对比。

工具：Applitools / Percy 智能：忽略抗锯齿微小差异 高亮：布局错位 / 元素缺失
🔌 Visual Regression Skill

7. 团队协作全景链路：殊途同归的工程闭环

无论起点在哪里，最终都会在 IDE 核心节点汇聚，产出工业级代码。

路径 A：需求/逻辑驱动

PRD 导入 IDE

产品经理或设计将结构化文档 + 界面截图喂给 Cursor。侧重于补全业务逻辑与流程分支。

⚠️ 前提：PRD 结构化规格

AI 依赖的是 Token 化后的语义关联而非人类直觉。模糊的自然语言需求会导致模型"吞咽困难"。走此路径前，PRD 必须完成以下结构化：

  • ▸规格驱动 — 用 Markdown/JSON 描述功能需求、非功能约束及边缘情况
  • ▸状态显式化 — 明确标注页面状态机（loading → success → error）
  • ▸API 契约先行 — 附带 OpenAPI 规格或标注接口路径、参数、返回类型
PRD Structuring Skill

可封装为 Skill — 自动将自然语言 PRD 转成结构化规格。

路径 B：视觉/规范驱动

Figma 设计稿导入 IDE

设计师通过插件提炼样式元数据 JSON 同步至 IDE。侧重于极致像素还原与品牌一致性。

⚠️ 前提：设计文件结构化治理

AI 阅读的是图层树而非像素。如果 Figma 图层树混乱，AI 生成的 DOM 结构也会随之坍塌。走此路径前，设计稿必须完成以下治理：

  • ▸语义化命名 — 使用 ProjectForm、HeaderNav 而非 Frame 74、Group 12
  • ▸Slot 弹性区域 — 利用 Figma Slot 标记可替换内容区
  • ▸图层树 ≈ DOM 树 — 保持视觉层级与代码结构一致
Figma-to-Token Skill

可封装为 Skill — 自动同步 Figma 改动到 IDE 代码。

💡 业务化工程落地的核心建议

🚫 摆脱"一句话生成应用"的幻想

这种低复杂度的生成仅适用于内部极简原型。真正的业务系统具有高度定制化需求。正如熟练的木匠使用电锯，AI 是提效工具，但业务逻辑和软件架构必须由人类掌握。

📝 提升"规格书编写"能力

未来的设计师需要掌握"语境工程"——学会通过 Markdown、JSON 或设计令牌来描述设计规则，而不是仅依赖感性描述。当提供清晰的"设计规格"时，AI 表现会显著提升。
