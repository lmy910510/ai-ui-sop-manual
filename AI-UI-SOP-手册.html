1. 现象分析：理想与现实的鸿沟

💭 理想主义 (Hype)

  • ❌ “一句话就能生成完整的商业应用。”
  • ❌ “AI能和设计师心灵相通，理解模糊的需求。”
  • ❌ “以后不需要前端开发和交互设计师了。”
  • ❌ 对着界面说“给我加个AI入口”，它就能完美契合业务流。

🧱 工程现实 (Reality)

  • ✅ AI本质是概率预测引擎，缺乏业务上下文，需要显示喂养。
  • ✅ 模糊的提示词必定产生随机的结果（Hallucination）。
  • ✅ 它的定位是“代码/组件装配工”，而非产品架构师。
  • ✅ 必须将感性设计转化为理性协议，AI 才能真正落地。

3. 能力边界分析：人机协同的界限

🤖 AI 核心优势：

极速原型生成：秒级产出讨论物料；标准组件装配：完美遵循组件库规范；多版本样式探索：基于风格迁移的海量方案出图。

👤 人类核心职责：

业务意图抽象：定义“为什么”；设计协议制定：输出 JSON/MD 物理规则；复杂逻辑闭环：API 绑定与状态管理；最终品控：审美最终决策。

4. 工业级标准化作业程序 (SOP)

将模糊对话转化为**机器可执行的协议**。这是确保 AI 产出“工程级代码”的唯一路径。

1

样式协议化 (Skeleton)

tokens.json

定义物理约束。建立三层令牌体系，消除 AI 的视觉盲猜。

📑 深度示例：语义化 Token 体系
{ "project": "TAI-6.0", "primitive": { "teal-500": "#0D9488", "teal-700": "#0F766E", "stone-900": "#1C1917" }, "semantic": { "background": { "main": "{primitive.stone-900}", "action": "{primitive.teal-500}", "action-hover": "{primitive.teal-700}" }, "text": { "on-action": "#FFFFFF", "muted": "#A8A29E" } }, "layout": { "radius": { "card": "24px", "btn": "12px" }, "spacing": { "md": "16px", "lg": "32px" } }, "component-constraints": { "primary-button": { "min-height": "64px", "font-weight": "700" } } }
2

意图规格书 (Soul)

spec.md

注入设计的灵魂。用结构化文档解释业务场景逻辑与“不可逾越的红线”。

📝 深度示例：包含状态机逻辑的交互规格书
# 场景：驾驶模式下的智能助手交互 ## 1. 核心触发逻辑 - **初始化**：组件以 `semantic.background.main` 色彩常驻于屏幕底栏 20% 处。 - **驾驶感应**：若检测到 `car_speed > 0`，强制进入“安全性门禁模式”： - 自动将按钮点击热区扩大至全宽。 - 禁止展示任何细颗粒度的辅助文本，仅保留核心 Action。 ## 2. 反馈闭环规范 - **监听中 (Listening)**：背景需执行 2.0s 周期的渐变呼吸效果。 - **思考中 (Thinking)**：展示 `tokens.json` 中定义的 `action` 颜色的 loading 状态。 - **错误边界**：若 AI 响应超时，展示警告图标并回退至上一级导航，不得在当前页死等。
3

封装参考库 (Reference)

.tsx / .jsx
核心资产库

这是存放你封装代码块的地方！ 喂入私有源码（Few-shot），AI 才能生成真正工程化的调用。

💻 深度示例：企业级封装组件源码 (带业务 Hook)
// 你已经写好的私有资产：TAI_BaseCard.tsx import { useDriverState } from '@/hooks/useDriverState'; import { Tokens } from '@/config/theme'; export const TAI_Container = ({ title, children, status = 'idle' }) => { const { isDrivingMode } = useDriverState(); return ( <div className={`p-6 flex flex-col gap-4 rounded-[--radius-card] shadow-xl ${isDrivingMode ? 'bg-secondary' : 'bg-primary'} ${status === 'error' ? 'border-red-500' : 'border-stone-800'}`}> <div className="flex justify-between items-center"> <h3 className="text-xl font-extrabold text-white">{title}</h3> <div className="status-indicator" /> </div> <div className="w-full h-px bg-stone-700 opacity-30" /> <div className="card-body-content"> {children} </div> </div> ); };
4

人机交接与交付 (QA Gate)

Delivery

明确工程边界：AI 产出“皮囊”，人类负责注入“数据活水”。

🤖 AI 产出的 80%

  • ✔ 复用 Step 3 里的私有封装组件进行装配
  • ✔ 自动填入 Step 1 里的语义化变量名
  • ✔ 生成响应式布局代码及 UI 状态展示逻辑
  • ✔ 遵循 Step 2 的交互场景物理约束

👤 人类介入的 20%

  • ✔ 后端 API 数据真实接口联调与错误分支捕获
  • ✔ 全局 Context/Redux 状态树同步与权限分发
  • ✔ 针对极细微交互细腻度的最后像素级微调
  • ✔ 性能优化及正式部署前的工程安全性审计

5. 团队协作全景链路：殊途同归的工程闭环

无论起点在哪里，最终都会在 IDE 核心节点汇聚，产出工业级代码。

路径 A：需求/逻辑驱动

PRD 导入 IDE

产品经理或设计将结构化文档 + 界面截图喂给 Cursor。侧重于补全业务逻辑与流程分支。

路径 B：视觉/规范驱动

Figma 规范导入 IDE

设计师通过插件提炼样式元数据 JSON 同步至 IDE。侧重于极致像素还原与品牌一致性。

↓ 殊途同归汇聚点 ↓

汇聚节点：AI Demo 生成 (IDE / Cursor)

AI 深度整合 **SOP 四项物料 (JSON + MD + 代码库 + API)**，吐出首版可运行页面。这是团队所有分支的汇聚点。

↓ 循环精修迭代 ↓

精修动作：如何让 Demo 达到设计预期？

1. 自然语言追加提示

“按钮太靠右了，参考 tokens.json 的间距系统向左移 16px，并提升主色饱和度。”

2. 引用代码块强制重构

“将此处的卡片组件替换为项目中的 `@TAI_Container.tsx` 封装组件。”

3. Figma 双向联动纠偏

局部视觉调不准时，在 Figma 精修后重新导出 JSON 覆盖 IDE 的主题定义。

↓ 工业落地交付 ↓

研发：工业级落地

研发接手已经“长相正确”且符合规范的 Demo 代码，完成最后的数据逻辑绑定、鉴权与闭环，正式转为生产代码。
